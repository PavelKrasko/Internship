# VM vs Контейнеры: сравнительный анализ
## Введение

Виртуализация и контейнеризация решают одну задачу — изоляцию рабочей среды —
но делают это принципиально разными способами. Понимание этой разницы критично
для выбора правильного инструмента в реальных проектах.
---

## 1. Время запуска

**Виртуальная машина (VM)** при запуске проходит полный цикл инициализации:
BIOS/UEFI, загрузчик, ядро гостевой ОС, системные демоны. Это занимает
**от 30 секунд до нескольких минут** — как при включении физического компьютера.

**Контейнер** не загружает ОС — он запускает **один изолированный процесс**
поверх уже работающего ядра хоста. Старт занимает **миллисекунды**.
`docker run nginx` выходит в рабочее состояние быстрее, чем успеет моргнуть
курсор в терминале.

> **Вывод:** контейнеры в 100–1000 раз быстрее запускаются, что критично
> для автомасштабирования и CI/CD пайплайнов.

---
## 2. Размер на диске

**VM** хранит полный образ диска гостевой ОС: ядро, системные библиотеки,
пакетный менеджер, утилиты. Образ Ubuntu VM — **~2–4 GB**, Windows — **~20–40 GB**.

**Контейнер** содержит только то, чего нет на хосте: файловую систему приложения
и необходимые пользовательские библиотеки. Образ `ubuntu` в Docker — **~30 MB**,
`alpine` (минималистичный Linux) — **~5 MB**. При этом общие слои образов (layers)
кешируются и не дублируются на диске.

> **Вывод:** контейнеры занимают в 50–100 раз меньше места благодаря
> отсутствию гостевого ядра и слоистой файловой системе (UnionFS).

---
## 3. Потребление RAM

**VM** резервирует память под гостевую ОС жёстко: если выделено 2 GB —
они заняты, даже если VM простаивает. Гипервизор (VirtualBox, VMware)
потребляет ресурсы поверх этого.

**Контейнер** использует ровно столько памяти, сколько нужно **процессу внутри**.
Nginx в контейнере потребляет ~5–10 MB RAM. Никакого оверхеда от гостевой ОС нет:
ядро одно — хостовое.

> **Вывод:** на одной машине можно запустить сотни контейнеров вместо
> десятков VM, не расходуя память впустую.

---
## 4. Уровень изоляции

**VM** даёт **аппаратную изоляцию**: гостевая ОС работает с виртуальным
железом через гипервизор. Уязвимость в гостевой ОС не компрометирует хост.
Это максимальный уровень безопасности — «побег» из VM практически невозможен.

**Контейнер** использует механизмы ядра Linux — **namespaces** (изоляция
процессов, сети, файловой системы) и **cgroups** (лимиты CPU/RAM). Изоляция
реальная, но ядро общее. Уязвимость на уровне ядра теоретически может
затронуть все контейнеры на хосте.

> **Вывод:** VM изолированы сильнее; контейнеры достаточны для большинства
> задач, но не подходят для запуска недоверенного кода без дополнительных мер
> (gVisor, Kata Containers).
